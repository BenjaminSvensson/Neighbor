# Codex Instructions

- This repository-local `AGENTS.md` file is the authoritative, always-on project instruction source for all Codex work in the Neighbor repository. Do not edit or override any global Codex `AGENTS.md` when changing Neighbor project instructions.
- At the start of every task, read this file, confirm the current branch with `git status --short --branch`, and update local `main` with `git switch main` followed by `git pull --ff-only origin main` before creating a task branch unless the user explicitly asks to stay on the current branch.
- Work on a separate task branch for Codex-authored changes unless the user explicitly asks to stay on the current branch. Branch from the latest `main` whenever possible, and use a clear `codex/<task-name>` branch name for routine Codex work.
- When Codex finishes making changes in this repository, commit on the task branch, switch back to `main`, update `main` with `git pull --ff-only origin main`, safely merge the task branch into `main`, push `main` to GitHub, and run `git lfs push --all origin main` before reporting completion.
- If the task branch cannot be cleanly merged into `main`, or if pulling `main` would overwrite local work, stop and explain the blocker and exact next step instead of forcing the merge.
- Because this Unity project depends on Git LFS objects for device parity, do not treat `Everything up-to-date` from `git push` as sufficient when Unity assets, scenes, terrain, NavMesh, generated assets, imported assets, textures, audio, models, or other LFS-tracked content may be involved. Always verify `git lfs push --all origin main` completes successfully after pushing `main`.
- On a receiving PC, after pulling `main`, run `git lfs install`, `git lfs pull`, and `git lfs checkout` before opening the Unity project. If Unity assets are still missing, check `git lfs status`, `git lfs ls-files`, and confirm the source PC ran `git lfs push --all origin main`.
- If a receiving PC still shows missing or stale terrain/NavMesh after the Git and LFS pull steps, close Unity first, then force-refresh the tracked project files with `git restore --source origin/main -- Assets ProjectSettings Packages`, run `git lfs checkout`, and reopen Unity. If the Git working tree is clean and file hashes match the source PC but Unity still renders old or missing terrain, delete the receiving PC's local `Library` folder so Unity performs a full reimport; `Library` is a generated cache and must not be committed.
- To diagnose cross-PC terrain mismatches, compare `Get-FileHash -Algorithm SHA256` output for the terrain data files, scene files, terrain layers, NavMesh assets, and terrain textures on both PCs before assuming GitHub is missing changes.
- Stage and commit only the files changed for the current task. Do not include unrelated user edits already present in the working tree.
- Unity scene files are an exception to the rule above. Before every commit, check for modified or untracked `Assets/**/*.unity` and matching `.unity.meta` files. Include all scene changes in the commit and push, even when they were manually authored or are unrelated to the current task, so project state is not left behind between PCs. Never discard, reset, stash, or overwrite scene changes without explicit user approval.
- Unity project-state assets are also an exception when they are needed for device parity. Before every commit, check for modified or untracked terrain, NavMesh, generated/imported asset folders, and their matching `.meta` files, especially `Assets/New Terrain*.asset`, `Assets/Main/Art/Terrain/Data/**/*.asset`, `Assets/**/NavMesh*.asset`, `Assets/**/*.terrainlayer`, and generated `Assets/**/Materials/` folders. Include these files in the commit and push when they represent the current Unity project state, even if they were manually authored or Unity-generated, so switching PCs produces the same project. If it is unclear whether a dirty Unity asset is intentional project state or unrelated work, stop and explain the exact files that need owner confirmation. Never discard, reset, stash, regenerate, or overwrite terrain/NavMesh/project-state assets without explicit user approval.
- Before reporting that there are no Unity project-state changes to publish, run `git status --short --untracked-files=all -- Assets`, `git lfs status`, and `git stash list`. If a stash appears to contain scene, terrain, NavMesh, material, generated asset, or imported asset changes, do not apply or drop it automatically; report the stash name and relevant files and ask the user whether it is intentional project state.
- Treat binary Unity `.asset` files such as TerrainData and NavMesh as authoritative binary project state. Do not convert them to YAML/text, do not normalize line endings, and keep `.gitattributes` rules in place so these assets remain binary-safe in Git.
- If pushing is blocked by authentication, failing checks, or unclear change ownership, explain the blocker and the exact next step needed.
